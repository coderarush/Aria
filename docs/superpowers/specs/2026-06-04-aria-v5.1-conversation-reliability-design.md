# Aria v5.1 — Conversation Reliability (talk-over barge-in + a bulletproof audio pipeline) — Design Spec

**Date:** 2026-06-04
**Status:** Approved (pending written-spec review)
**Builds on:** Aria v4 (autonomous agent; Gemini-only voice)

## Summary

v5.1 makes *talking to Aria* feel premium. It rebuilds the audio pipeline around a
single engine with a real **acoustic echo canceller (AEC)** so Aria can be
**interrupted mid-sentence** (full talk-over barge-in — the JARVIS moment), and —
as a byproduct of owning the pipeline cleanly — it also **never goes deaf**, the
**wake word fires every time**, responses feel **faster**, and it **mis-hears
less**. All of it runs **100% locally and free** (the AEC is a vendored C library,
no API, no quota), preserving the project's free-tier guarantee.

## Roadmap context

v5 = **"reliability + depth"** (make what exists flawless before the generic
computer-use leap, which is v6). v5 decomposes into four sequenced pillars, each
its own spec → plan → ship increment:

1. **Conversation reliability** *(this spec — v5.1)*: talk-over barge-in + a
   bulletproof audio/wake pipeline.
2. **Cross-session memory** (v5.2): remembers you, your context, past tasks.
3. **First-class integrations** (v5.3): Calendar/Reminders/Mail/Notes/Music via
   real APIs (EventKit etc.), not fragile AppleScript.
4. **Speaker verification** (v5.4): only obeys your voice (on-device voiceprint);
   builds on this pillar's audio pipeline.

End vision unchanged: personal JARVIS, eventually a sellable product.

## Decisions (locked with user)

| Topic | Decision |
|-------|----------|
| Barge-in | **Full talk-over** — interrupt her any time you speak |
| Echo cancellation | **Vendored WebRTC-class AEC library** (`webrtc-audio-processing` APM or `speexdsp` echo canceller — an early milestone picks the one that vendors cleanest into SwiftPM). Both are free/BSD. |
| Architecture | **Rebuild the audio pipeline (Approach A)**: one `AVAudioEngine` owns mic + TTS playback; AEC'd mic buffers are fed manually to the recognizer. The prior `setVoiceProcessingEnabled` failures came from the recognizer fighting its own audio path — owning one engine + feeding buffers manually fixes that. |
| De-risk | An **AEC-validation milestone first** (prove cancellation on this hardware before building the pillar on top of it). |
| Also fix | Wake fires every time · never goes deaf · faster responses · doesn't mis-hear |
| Free tier | AEC is **fully local** — no API calls, zero quota impact. |

## Non-Goals (out of v5.1)

- Cross-session memory, app integrations, speaker verification → v5.2–v5.4.
- Generic computer use (vision/click any app) → v6.
- Multi-speaker diarization beyond the v5.4 owner voiceprint.
- Wake-word model replacement (we keep SFSpeech-based wake; only the audio feeding
  it changes). A custom keyword model is a later option if accuracy demands it.

## Section 1 — Architecture & components

The mic is currently owned implicitly by `WakeWordEngine` (SFSpeechRecognizer with
its own audio path), and TTS plays through a standalone `AVAudioPlayer`. Those two
never share a graph, so the recognizer always hears Aria's own voice — which is
why barge-in was impossible and self-trigger was a constant threat. v5.1 unifies
them.

**New components**

- **`AudioBus`** (Core/Audio) — the single owner of audio I/O. One `AVAudioEngine`
  with: the mic **input node** (tapped), and an **`AVAudioPlayerNode`** that plays
  Aria's TTS. It resamples mic + TTS to the AEC working rate (mono 16 kHz) and
  emits time-aligned **(near-end mic, far-end TTS) frame pairs** to the echo
  canceller, then publishes the **cleaned near-end** stream to consumers. Exposes
  `playReference(pcm:)` (play Aria's voice AND register it as the AEC far-end) and
  `stopPlayback()` (instant, for barge-in).

- **`EchoCanceller`** (Core/Audio) — a thin Swift wrapper over the vendored C AEC
  library. API: `process(near: [Int16], far: [Int16]) -> [Int16]` (cleaned
  near-end). Owns the library's state/handle, the analog/adaptive filter, and the
  delay estimate. Pure/synchronous; unit-testable with fixture WAVs.

- **`WakeWordEngine` v2** (Core) — keeps the wake/command state machine but no
  longer owns audio. It consumes the **cleaned** buffers from `AudioBus` and feeds
  them to `SFSpeechRecognizer` via `recognitionRequest.append(buffer)`. Because the
  audio engine is decoupled from the recognizer lifecycle, recognizer restarts no
  longer risk the mic dying — the foundation of "never deaf."

- **`BargeController`** (Core) — runs a VAD on the cleaned near-end. When Aria is
  speaking (TTS active) AND sustained user speech is detected in the cleaned stream
  (energy above threshold for ~150–250 ms), it fires `onBarge`: stop TTS, cancel
  the in-flight turn, and let the barge utterance flow on as the next command (no
  re-wake).

**Changed/extended (v4 integration)**

- **`VoiceEngine`** — instead of its own `AVAudioPlayer`, it hands TTS PCM to
  `AudioBus.playReference(pcm:)`. `stop()` → `AudioBus.stopPlayback()`. The
  completion callbacks (`onFinish`/`onChunkFinished`) now fire from `AudioBus`
  playback completion. (Gemini-only voice is unchanged.)
- **`AriaController`** — wires `BargeController.onBarge` (replacing the disabled
  `onSpeechOnset`): cancel `currentTurnTask`, stop TTS, capture the new command.
  The existing `bargeInEnabled` / `bargeInSensitivity` settings become real.
- **`StreamingVoice`** — unchanged interface; its `speakChunk` now routes through
  `AudioBus`.

## Section 2 — The AEC + barge-in pipeline

**The reference signal.** AEC needs the *far-end* (what's played to the speaker)
time-aligned with the *near-end* (mic). We hold Aria's exact TTS PCM (24 kHz mono
from Gemini). Because TTS plays through `AudioBus`'s player node, playback timing
is known: we feed the scheduled/rendered TTS frames as the far-end, and the AEC
library's **delay estimator** (WebRTC APM) or a configured **filter tail length**
(Speex, ~150–250 ms) absorbs the residual acoustic delay. No external clock needed.

**Sample-rate handling.** Mic input (typically 44.1/48 kHz) and TTS (24 kHz) are
both converted to the AEC working rate (mono 16 kHz) via `AVAudioConverter`. The
cleaned 16 kHz near-end is what feeds SFSpeech (converted to the recognizer's
expected format on `append`). Frame size matches the library (e.g. 10 ms / 160
samples).

**Data flow.**
```
mic ─► AudioBus tap ─► resample 16k ─┐
                                     ├─► EchoCanceller.process(near, far) ─► cleaned 16k ─┬─► SFSpeech (append)
TTS PCM ─► player node ─► resample 16k ┘ (far-end reference)                              └─► BargeController VAD
```

**Interrupt flow.** While `VoiceEngine` is speaking, the recognizer is *kept
listening on the cleaned stream*. Two outcomes:
- VAD detects sustained user speech → `BargeController.onBarge` → `AudioBus.stopPlayback()`
  + cancel turn → the recognizer's in-progress transcript becomes the next command.
- No speech → Aria finishes; normal turn end (the v4 re-arm path, unchanged).

Self-trigger (mic hearing Aria) is *structurally* solved: the recognizer only ever
sees the cleaned stream, where Aria's voice has been cancelled.

## Section 3 — Reliability hardening (wake + never-deaf)

- **Single long-lived audio engine.** `AudioBus`'s `AVAudioEngine` starts once and
  stays running. Recognizer churn (SFSpeech ends sessions on silence / errors)
  cycles only the `SFSpeechRecognitionRequest` + recognition `Task`, never the mic.
  This removes the class of bug where a recognizer restart killed audio capture.

- **Guarded restart + watchdog.** `WakeWordEngine` v2 keeps the v1 `sessionID`
  guard (a superseded recognition task's completion can't schedule a restart). Adds
  a **watchdog**: if no cleaned buffers arrive for N ms, or the recognizer errors,
  it tears down and re-creates the request/task (guarded) while the engine keeps
  feeding audio. After any turn/task it returns to a clean wake state (the v4
  `endConversation` contract — `isSuspended=false`, `mode=.wake`).

- **Wake every time.** Continuous cleaned audio into the wake detector (no gaps
  from audio restarts) + a tuned partial-result match for "hey aria" with a small
  confusable set ("hey area", "hey arya"). Sensitivity exposed in Settings.

## Section 4 — Latency & accuracy

- **Latency.** The mic is always live (no wake→listen audio spin-up), so wake→
  listening is immediate. First-spoken-word latency is already low (v3 streaming +
  Gemini once); v5.1 trims the wake→capture handoff and avoids the old
  recognition-restart stalls.
- **Accuracy / endpointing.** Use SFSpeech partial results + a short trailing-
  silence endpoint so she knows when you're done without cutting you off. The
  cleaned stream (no self-audio) reduces mis-transcription during/after her speech.

## Section 5 — Dependencies & free-tier

- **AEC library** is vendored as a SwiftPM **C target** (source + module map) with
  a Swift wrapper. `speexdsp`'s echo canceller is plain C and vendors easily;
  `webrtc-audio-processing` (APM, includes AEC3) is higher quality but autotools-
  based and harder to package — **Milestone 0 picks the one that builds cleanly**
  for this app. Both are free (BSD-style).
- **Zero API cost.** AEC, VAD, resampling, and wake all run on-device. No Gemini
  calls added — the free-tier guarantee is untouched. (TTS still uses the Gemini
  voice as in v4.)

## Section 6 — Edge cases & testing

**Edge cases:** no echo path (headphones) → AEC is a near-passthrough, still fine;
very loud room → VAD threshold tuned, false-barge avoided by the sustained-speech
window; recognizer error mid-barge → watchdog restart; TTS stopped mid-utterance →
player node flush + clean re-arm; AEC delay drift → APM delay estimator / Speech
tail length absorbs it.

**Testing:**
- **Unit:** `EchoCanceller` — fixture WAVs (near = speech + echo, far = echo
  source) → assert residual echo energy reduced ≥ target dB and cleaned correlates
  with clean speech; `BargeController` VAD onset (speech vs silence vs Aria-only);
  `WakeWordEngine` v2 restart/never-deaf state machine (simulated recognizer error
  → guarded restart; watchdog fires on buffer starvation); resampler frame math.
- **Manual smoke:** live talk-over (interrupt her, she stops, your words are
  captured); wake reliability across 20+ cold tries; never-deaf across 20+ turns
  and a task; latency feel; noisy-room accuracy.
- Keep **all v4 tests green** + add the above.

## Section 7 — Suggested implementation staging (for the plan)

0. **AEC validation (de-risk first).** Vendor the chosen lib; standalone test:
   feed near/far fixture WAVs, confirm cancellation on this hardware. Pick
   speexdsp vs webrtc-apm here.
1. **`AudioBus`** — one engine, mic tap + player node, resampling, time-aligned
   (near, far) frame pairs; `playReference`/`stopPlayback`.
2. **`EchoCanceller`** wired into `AudioBus` → publish the cleaned stream (TDD with
   fixtures).
3. **`WakeWordEngine` v2** — consume cleaned buffers via `append`; guarded restart
   + watchdog (never-deaf); wake + command modes.
4. **`BargeController`** + wiring — talk-over interrupt in `AriaController`; route
   `VoiceEngine` TTS through `AudioBus`; make `bargeInEnabled` real.
5. **Latency + accuracy** — endpointing, partial results, handoff trim.
6. **Settings + polish** — barge-in sensitivity, full v4 regression, manual smoke.

## Risks / call-outs

- **AEC integration is the hard part** — vendoring the C lib into SwiftPM + time-
  aligning the far-end reference. Milestone 0 de-risks it before the pillar is
  built on top.
- **SFSpeech with manually-fed buffers** must accept our format — validated in
  Milestone 3 (it supports `append(_:)` of `AVAudioPCMBuffer`).
- **Resampling overhead** adds a little CPU; negligible at 16 kHz mono, validated
  for latency in Milestone 5.
- **Hardware variance** — AEC tuning may differ across Macs/mics; thresholds and
  filter length are configurable, validated on the user's machine.
